# Тема 1: Мобильное приложение для заказа доставки еды  

## Описание информационной системы

Мобильное приложение для заказа доставки еды — это решение для смартфонов, позволяющее пользователям выбирать блюда из меню ресторанов-партнеров, оформлять заказы, отслеживать их статус в реальном времени и оплачивать онлайн. Система включает:

1. Мобильное приложение (iOS/Android) с личным кабинетом пользователя.  
2. Серверную часть для обработки заказов, управления меню и взаимодействия с курьерами.  
3. Интеграцию с платежными системами (например, Сбербанк, QIWI) для онлайн-оплаты.  
4. Базу данных для хранения информации о пользователях, заказах и предпочтениях.
5. Систему уведомлений (SMS, push-сообщения) о статусе заказа.

Доступ к приложению осуществляется через регистрацию с использованием номера телефона или email. Данные передаются по зашифрованным каналам (HTTPS), а оплата производится через защищенные платежные шлюзы.  

## Задание 1. Анализ информационной системы

Тип обрабатываемой информации:

1. Персональные данные пользователей (ФИО, номер телефона, email).

2. Адреса доставки.

3. История заказов (блюда, стоимость, время).

4. Платежные данные (реквизиты карт при сохранении для быстрой оплаты).

5. Геолокационные данные (для отслеживания курьера).

Категория персональных данных:

| Вид данных | Категория ПД |
| ---------- | ------------- |
| ФИО, телефон, email | Обыкновенные персональные данные |
| Адрес доставки | Обыкновенные персональные данные |
| Реквизиты банковских карт | Сведения, требующие повышенной защиты |
| История заказов | Не персональные данные |

Возможные каналы утечки информации:

1. Утечка базы данных из-за слабой защиты сервера (например, отсутствие шифрования).

2. Фишинговые атаки: поддельное приложение в магазине приложений собирает данные пользователей.

3. Перехват платежных данных при передаче из-за уязвимостей в интеграции с платежными системами.

4. Несанкционированный доступ к аккаунту через подбор пароля или использование общего устройства.

5. Утечка геоданных курьеров и клиентов через ошибки в настройках API.

## Задание 2. Выявление угроз

| № | Угроза | Источник угрозы | Последствия | Вероятность |
| - | ------ | --------------- | ----------- | ----------- |
| 1 | Кража платежных данных | Внешние хакеры | Финансовые потери пользователей, репутационный ущерб | Высокая |
| 2 | Утечка базы пользователей | Взлом сервера | Спам-рассылки, мошенничество с данными | Средняя |
| 3 | Подделка приложения в App Store | Мошенники | Массовое хищение логинов и паролей | Высокая |
| 4 | Перехват геолокации курьеров | Злоумышленник в локальной сети | Слежка за курьерами, кража заказов | Низкая |

## Задание 3. Формирование требований к защите информации

1. Требование к шифрованию:

- Все данные пользователей (включая платежные реквизиты) должны шифроваться при передаче (TLS 1.3) и хранении (AES-256).

- Запретить сохранение полных данных банковских карт — использовать токенизацию через платежные системы.

2. Требование к аутентификации:

- Внедрить двухфакторную аутентификацию (SMS-код или биометрия) для входа в аккаунт.

- Обязательная смена пароля при подозрении на компрометацию.

3. Требование к безопасности платежей:

- Проводить регулярные аудиты интеграции с платежными шлюзами на соответствие стандарту PCI DSS.

- Отказаться от хранения CVV-кодов карт даже в зашифрованном виде.

4. Требование к защите:

- Разместить официальные ссылки на приложение только на сайте компании.

- Добавить в приложение уведомление о поддельных версиях в магазинах приложений.

5. Требование к минимизации данных:

- Автоматически удалять историю заказов старше 6 месяцев, если это не требуется для законодательства.

- Не запрашивать геолокацию после завершения доставки заказа.

## Вывод

Анализ показал, что мобильное приложение для доставки еды обрабатывает данные, требующие защиты от кражи и утечек. Основные риски — компрометация платежных данных и персональной информации пользователей. Для снижения угроз необходимо внедрить шифрование, многофакторную аутентификацию и регулярные проверки безопасности. Также важно минимизировать сбор данных и обучать пользователей распознавать фишинг. Соблюдение этих требований позволит соответствовать законодательству и сохранить доверие клиентов.
